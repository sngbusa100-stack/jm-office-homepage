- 작성일: 2026-07-09
- 상태: 사용자 4개 섹션(구조·진단 흐름·콘텐츠 원칙·기술 검증) 승인 완료
- 프로젝트 위치: `D:\홈페이지\정명 행정사사무소 홈페이지`
- 참고 기준: `D:\홈페이지\행정사 비자진단 홈페이지` (이하 "비자진단 사이트")

## 1. 배경과 목표

기존 `D:\홈페이지` 루트의 정적 홈페이지(index.html, styles.css, script.js)는
가짜 연락처·가상 성공 후기·목업 챗봇·동작하지 않는 상담 폼으로 구성된 시안 수준이다.
방문자가 궁금증을 해결할 콘텐츠도, 수임을 신청할 실제 장치도 없다.

목표:

1. 방문자가 스스로 상황을 점검하고(셀프 진단), 행정사의 필요성을 이해하고(전문 콘텐츠),
   상담·수임을 신청(3중 채널)하는 흐름을 완성한다.
2. 비자진단 사이트와 동일한 품질 기준(정직성, 접근성, 자동 테스트)을 적용한다.
3. 개업 준비 중 상태를 모든 주요 화면에 명시하고, 개업 후 값만 채우면
   실제 접수가 활성화되는 구조를 만든다.

## 2. 확정된 사용자 결정

- 개업 상태: 아직 개업 준비 중. 실제 연락처·등록번호 없음.
  → 개업 준비 중 표시 유지, 수임 흐름은 완성하되 연락처는 개업 후 채우는 구조.
- 구축 방식: Vite + React + TypeScript 재구축. 출입국 분야는 비자진단 사이트로 연계.
- 수임 채널: ① 전화 + 카카오톡 채널(자리 마련) ② 온라인 상담 신청 폼
  ③ 사건 유형별 셀프 진단 후 접수 — 3가지 모두.
- 진단 범위: 주력 3분야(음주운전 구제, 영업정지 구제, 국가유공자 등록) 정밀 진단.
  나머지 분야는 상세 가이드 + FAQ + 상담 유도.
- 언어: 한국어 전용. i18n 계층 제외.

## 4. 진단 → 수임 흐름

### 4.1 진단 문항

분야별 8~12문항. 답변은 `sessionStorage`에만 저장하고 서버로 전송하지 않는다.

- 음주운전: 적발 일시, 혈중알코올농도 구간, 처분 유형(정지/취소), 처분 통지 수령일,
  운전 사유(생계형 여부), 과거 이력, 이의제기·행정심판 기한 경과 여부 등
- 영업정지: 업종, 처분 사유, 처분 통지 수령일, 영업 규모·생계 의존도,
  사전통지·의견제출 절차 진행 상황, 기한 경과 여부 등
- 국가유공자: 신청 유형, 복무·공무 관련성, 상이·질병 기록 보유 여부,
  과거 신청·기각 이력, 재심의 기한 등

### 4.2 결과 분류 (비자진단과 동일 원칙)

A~D 등급·백분율·"구제 가능성" 표현을 사용하지 않는다. 결과는 네 수준으로 분류한다.

1. 확인됨 — 답변 기준으로 준비된 항목
2. 서류 보완 — 준비·보완이 필요한 서류 항목
3. 공식 확인 — 관할 기관·공식 출처에서 확인이 필요한 항목
4. 긴급 확인 — 법정 기한이 임박했거나 경과 우려가 있는 항목

긴급 확인 항목에는 근거 법령과 기한을 함께 표시한다.
예: "행정심판 청구는 처분이 있음을 안 날부터 90일 이내입니다(행정심판법 제27조)."

### 4.3 상담 연결

- 결과 화면 하단에 "이 진단 결과로 상담 신청" 버튼 → `/consult`로 이동하며
  결과 요약(분야, 분류별 항목 수)을 sessionStorage로 전달해 폼에 미리 채운다.
- 개업 전에는 폼 제출 대신 "개업 준비 중 — 상담 접수는 개업 후 시작됩니다" 안내를 표시한다.
- 긴급 확인이 있는 결과에는 공식 채널(국민신문고, 중앙행정심판위원회 110,
  관할 기관)을 함께 안내해 개업을 기다리다 기한을 놓치지 않게 한다.

### 4.4 개업 후 활성화 구조

사무소 정보를 단일 설정 파일 `src/data/office.ts`로 관리한다.

```ts
export const office = {
  isOpen: false,            // 개업 후 true로 변경
  phone: null,              // 예: '02-000-0000'
  kakaoChannelUrl: null,    // 예: 'https://pf.kakao.com/...'
  address: null,
  registrationNumber: null, // 행정사 등록번호
  businessNumber: null,     // 사업자등록번호
  formEndpoint: null,       // Formspree 등 폼 수신 주소
};
```

- `isOpen: false`이면: 전화·카톡 버튼과 폼은 "개업 후 제공" 상태로 렌더링,
  개업 준비 중 배지를 전 화면에 표시, 푸터에 등록번호 자리 대신 준비 중 문구.
- `isOpen: true` + 각 값 입력이면: 실제 버튼·폼 활성화.
- 값이 채워지지 않은 상태에서 isOpen을 켜면 빌드 검증 스크립트가 실패하도록 한다.

## 5. 콘텐츠·법적 원칙

1. 가상 성공 후기 4건 전면 삭제. 실제 후기로 오인될 수 있는 어떤 형식(별점, 날짜,
   "OO님")도 사용하지 않는다. 대신 "이런 상황을 도와드립니다" 시나리오 카드로 대체한다.
2. 가짜 전화번호·주소·사업자등록번호·이메일·SNS 링크 전부 제거. `office.ts`만이
   사무소 정보의 유일한 출처다.
3. 목업 챗봇 삭제. FAQ를 분야별로 충실히 작성해 대체한다.
4. "100% 구제", "확실히", "보장" 등 결과를 단정하는 표현 금지. 자동 테스트로
   금지 표현이 콘텐츠에 없는지 검사한다.
5. 법정 기한·요건은 근거 법령 조항과 공식 출처 링크, 확인일을 함께 표시한다.
   (도로교통법, 행정심판법, 식품위생법, 국가유공자 등 예우 및 지원에 관한 법률 등)
6. 진단 결과·모든 안내에 "일반 정보이며 법률 자문이 아님, 최종 판단은 관할 기관"을 명시한다.
7. 행정사법 제21조의2(과장 광고 금지)와 표시광고법을 준수한다.

### 행정사 필요성 콘텐츠 (`/why`) 구성

- 행정사가 하는 일: 행정사법 제2조 업무 범위를 쉬운 말로 풀어 설명
- 언제 행정사가 필요한가: 서류 반려 반복, 기한 임박, 절차를 모를 때, 처분 불복
- 변호사·법무사·노무사와의 차이: 역할과 일반적 비용 수준 비교(구체 금액 단정 금지)
- 혼자 진행 vs 전문가 조력: 시간·반려 위험·기한 관리 관점 비교표
- 잘못 알려진 오해: "행정사는 대서방이다" 등 오해 바로잡기

## 6. 기술 구조

- 스택: Vite + React + TypeScript + React Router + Vitest (비자진단 사이트와 동일 버전대)
- 스타일: 비자진단 사이트의 `tokens.css` 디자인 토큰을 복사·재사용하되,
  본 사이트 브랜드(정명 로고, 남색·금색 계열 기존 팔레트)에 맞게 조정
- 상태: 진단 답변 sessionStorage, 그 외 서버·외부 전송 없음
- 이미지: 기존 `images/` 로고 재사용, 히어로·분야 일러스트는 SVG 제작
- 배포 대비: `public/_redirects`(Netlify)·`vercel.json`(Vercel) SPA 재작성 설정 포함
- Windows 빌드: 비자진단에서 검증된 `scripts/clean-dist` prebuild 방식 재사용
